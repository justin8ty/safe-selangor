**Product Requirements Document (PRD) — Crime Social + District Choropleth (Hackathon MVP)**

**1) Product Summary**
A web app for visualizing crime trends by district/city using a 2D Mapbox choropleth, combined from (1) official dataset, (2) community “live” reports, and (3) scraped sources (already in DB). Users can submit photo-backed crime reports; AI (vision + text) moderates and assigns a confidence score to auto-approve or route to a moderator. A live “twitch-like” feed shows recent approved reports, with likes/views and karma points.

**2) Goals**
- Provide fast, district-level situational awareness with clear crime rate levels on a map.
- Enable community reporting while reducing spam/low-quality submissions using AI moderation + moderator fallback.
- Show simple numeric trends (graphs) and lightweight engagement (likes/views/karma).

**Non-goals (Hackathon)**
- No scraping implementation (scraped data already normalized in DB).
- No PII detection/redaction logic.
- No external “existence check” or verification beyond moderation.
- No badges (gamification is karma-only).
- Not an emergency service.

**3) Target Users**
- Residents/commuters: check district crime levels and recent reports.
- Contributors: submit reports, gain karma.
- Moderators: review low-confidence submissions quickly.

**4) Data Sources & Schema**
All sources are already stored in a database with identical columns:
- `state`, `district`, `category`, `type`, `date`, `crimes` (integer count)

Sources:
- Official dataset (manually downloaded → DB)
- Scraped sources (manually converted → DB; no scraping code)
- Live reports (generated by this app; should map into same schema for aggregation)

**5) Core UX / Pages**
- **Map (Home)**
  - 2D Mapbox choropleth: district/city polygons colored by crime rate level.
  - Filters: state, date range, category, type, source (official/live/scraped/combined).
  - Hover/click district: tooltip + details panel (breakdown by category/type, source contributions, trend mini-chart).
- **Live Feed (Sidebar)**
  - Twitch-like scroll of newly approved user reports (district-only location display).
  - Each item: type/category, district, timestamp, likes/views, optional image thumbnail.
- **Report Detail**
  - Report info + attachment(s), status, likes/views, district.
- **Submit Report**
  - Email-login gated.
  - Required: exact location permission, at least 1 image attachment, category/type, date/time.
  - Optional: description text.
- **Moderator Console (MVP-light)**
  - Queue of `needs_moderator` reports with AI confidence + explanation, image preview, and approve/reject actions.

**Public Location Display Policy**
- Collect exact GPS for district assignment and internal audit.
- Display district-only publicly (no exact coordinates/pin for other users).

**6) Report Submission Requirements**
- **Location**
  - Prompt for exact location permission (GPS).
  - App assigns `district` (and `state`) via polygon containment / reverse geocode.
- **Attachment (Required)**
  - Minimum 1 image; allowed formats: `jpg/jpeg/png/webp`.
  - Basic constraints: size limit and max count (pick hackathon-friendly defaults).
- **Fields**
  - `category`, `type`, `date` (defaults to now), `description` (optional).
- **AI Assistance**
  - Autocomplete / suggest `category` and `type` from description (optional path).
  - Post-submit moderation uses image + text.

**7) AI Moderation Requirements (Vision + Text)**
Purpose: decide whether the submitted image plausibly depicts a crime/incident scene and whether the selected type/category is plausible.

Outputs (stored with the report):
- `decision`: `approved | rejected | needs_moderator`
- `confidence`: 0–100
- `reason_codes`: short machine-readable tags
- `explanation`: short text for moderators (and optionally user-friendly snippet)

Decision thresholds:
- `confidence >= 90` → auto-approve
- `confidence < 90` → route to moderator queue
- Optional hard reject when confidence is high for “not an incident / irrelevant image / spam” (still within your moderation scope; no PII logic)

User feedback:
- If rejected: show brief reason category (“Image unclear / unrelated / insufficient evidence”) and allow resubmission.

**8) Aggregation & Choropleth Scoring**
The choropleth shows a district score for the selected filters and time window.

Source weighting (given):
- 50% official
- 30% live reports
- 20% scraped

Computation (MVP approach):
- For each district and selected window, compute `crimes_official`, `crimes_live`, `crimes_scraped`.
- Normalize each source to comparable scale for the selected scope (e.g., min-max per state within the current date range).
- Combined score: `score = 0.5*N(official) + 0.3*N(live) + 0.2*N(scraped)`
- Map to discrete “crime rate levels” (e.g., 5 buckets) for coloring + legend.

Notes:
- Live reports need mapping to the same schema: each approved report increments `crimes` for its `state/district/category/type/date`.

**9) Engagement & Karma**
- Track per-report:
  - `views` (basic client event; de-dupe per user/session)
  - `likes` (1 per user)
- Karma (simple):
  - +K for approved report
  - -k (optional) for rejected report, or only after repeated rejections
  - Optional small bonus for likes milestones (keep simple)

**10) System Design Requirements**
Must use:
- **Pub/Sub pattern**
  - Events/topics (example):
    - `report.submitted`
    - `report.media_uploaded`
    - `report.moderation_completed`
    - `report.needs_moderator`
    - `report.approved`
    - `report.rejected`
    - `report.published`
    - `district.aggregates_updated`
    - `report.error`
- **Chain of Responsibility pattern** for processing pipeline
  - Handler 1: validate/auth/rate-limit (simple)
  - Handler 2: media validation + storage record
  - Handler 3: district assignment from exact GPS
  - Handler 4: AI moderation (vision+text) + confidence
  - Handler 5: decision routing (auto-approve vs moderator)
  - Handler 6: publish approved report (feed + aggregates) + notify subscribers
  - Handler 7: error/status handler (idempotency, retry, dead-letter, state transition guard)

Caching (in-memory):
- Recent approved reports for live feed.
- District aggregates for quick choropleth render.

**11) Authentication (Email-Based Login)**
- Very simple email login (recommended: passwordless OTP or magic link).
- Store: user id, email, karma, timestamps.
- Enforce basic rate limits on report submission per account.

**12) Storage / Data Model (MVP)**
- `users`: id, email, karma, created_at, last_login
- `reports`: id, user_id, state, district, category, type, date, description, status, ai_confidence, ai_reason_codes, ai_explanation, created_at
- `report_media`: id, report_id, storage_key/url, mime, size, created_at
- `moderation_actions`: id, report_id, moderator_id, action, notes, created_at
- `report_metrics`: report_id, views, likes
- Aggregated tables (or materialized views):
  - `district_daily_aggregates`: state, district, date, category/type breakdowns, source breakdowns, totals, normalized scores

**13) Non-Functional Requirements**
- Performance: district choropleth renders quickly for common filters; feed updates within a few seconds after approval.
- Reliability: pipeline is idempotent (no double-publish on retries).
- Security: image access via signed URLs or protected endpoints; minimal auth/session hardening.
- Privacy: exact GPS stored internally; district-only shown publicly.

**14) MVP Success Metrics (Demo)**
- End-to-end flow works: email login → submit report with image → AI decision → publish or moderator queue.
- AI routing works: auto-approves ≥90; moderator can approve/reject quickly.
- Choropleth updates and reflects combined weighting + filters.
- Live feed shows newly approved reports with likes/views and karma changes.
- Trend graphs render correctly from aggregates.
